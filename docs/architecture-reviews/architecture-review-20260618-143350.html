Problem

build_faction is shallow — interface (one function, six parameters) is nearly as complex as the implementation (fetch, thumbnail, format, link, embed, save).

Solution

One seam between ImagePipeline and WorksheetBuilder. The seam carries a manifest dict — no PIL, no xlsx, no requests on either side of it.

Wins

  • · locality: network errors surface at the pipeline boundary, not mid-row
  • · leverage: test WorksheetBuilder with a fake manifest — no network, no disk
  • · test ImagePipeline independently with fixture files
  • · run pipeline once, rebuild xlsx N times

2. Create a FactionRegistry seam — validate on load, not in the loop

Strong in-process
factions.py · build.py

Before — data leaks raw into caller

factions.py — FACTIONS dict (1,309 lines)
↓ raw dict, no contract
build_faction — FACTIONS[key] with implicit shape
KeyError if key missing · TypeError if "products" missing
loop — (name, price, slug, img_file) tuple unpacking
ValueError if tuple has wrong length · silent xlsx corruption
xlsx output

Every layer assumes the dict shape is correct. Failures are late and opaque.

After — seam validates at load time

FactionRegistry.load(key) → Faction
validates schema · raises FactionNotFoundError early
Faction dataclass — title, output, products: list[Product]
typed, validated · Product has name, price, slug, image_file
WorksheetBuilder.write(faction: Faction, …)
only receives valid data — no defensive checks needed
xlsx output

Errors fail at the registry boundary, not mid-row. Build logic is simplified.

Problem

factions.py has no interface — it exports a raw dict. Callers know its shape implicitly. No seam exists between "data" and "build" — errors propagate to the worst possible moment (mid-loop, mid-xlsx-write).

Solution

A FactionRegistry module with a load(key) interface returning a validated Faction dataclass. Validation at load time; downstream code gets clean types.

Wins

  • · locality: all schema errors surface at one place
  • · leverage: build logic drops all defensive dict-key checks
  • · test registry in isolation with fixture dicts
  • · the seam justifies typed tests for Faction

Deletion test passes: removing scraper.py concentrates the complexity back into one canonical place rather than spreading it.

Problem

scraper.py is a dead adapter — 305 lines of async Playwright code that duplicates image logic from build.py and is never called. One adapter justifies only a hypothetical seam; two adapters that never share a caller justify nothing.

Solution

Delete scraper.py. Document the manual extraction workflow (browser injection → factions.py) in AGENTS.md or a WORKFLOW.md. If a real seam ever emerges (e.g., scraper runs in CI), the two adapters are justified at that point.

Wins

  • · locality: image download logic in one place
  • · interface surface of the codebase shrinks
  • · no divergence risk between two implementations
  • · the manual process becomes the canonical workflow

4. Introduce game-system hierarchy — stop leaking across a missing seam

Speculative in-process
factions.py

Before — flat namespace, leaking game boundaries

FACTIONS = { }

stormcast_eternals
flesh_eater_courts
space_marines
chaos_space_marines
hh_vehicles
gloomspite_gitz
necrons
hh_infantry
AoS 40k Horus Heresy ← mixed, no seam

"Build all AoS" means filtering by output path prefix — knowledge of the output path leaks into the caller.

After — game systems as first-class modules

GameSystem.AOS

stormcast_eternals flesh_eater_courts gloomspite_gitz

GameSystem.W40K

space_marines chaos_space_marines necrons

GameSystem.HORUS_HERESY

hh_vehicles hh_infantry

Game system is the seam. Callers query by system, not by path prefix. Adding a new system is additive.

Problem

Three game systems share one flat namespace. There's no seam between them. Callers who want "all AoS" must filter by output path prefix — the directory layout leaks into build logic.

Solution

Nest factions under a GameSystem key. FactionRegistry.load(system, key) makes the seam explicit. Speculative because the current single-faction CLI doesn't need the hierarchy yet.

Wins

  • · locality: game-system rules live in one place
  • · leverage: build --system aos works without path knowledge
  • · adding a new game system is additive, not editorial

Top recommendation

Start with Candidate 1 — deepen build_faction

The image pipeline seam is the highest-leverage change: it splits a shallow, untestable function into two deep modules without touching the data layer. Once the seam exists, Candidate 2 (FactionRegistry) slots in naturally as the module that feeds the pipeline. Candidates 3 and 4 are cleanup that follows from the structure Candidates 1 and 2 establish.
